# Supabase 邮箱验证码配置说明

> 归档说明
>
> 本文档对应的是仁迈早期网页公开版的邮箱注册 / 验证码方案。
> 当前桌面版主产品不再以注册登录、邮箱验证和公开聊天为核心流程。
> 仅在你后续明确恢复旧网页版时再参考本文；当前桌面版实施请以本仓库中的桌面文档为准。

仁迈网页端现在已经接入了两条邮箱验证码流程：

- 注册公开账号时，必须先完成邮箱验证码验证才能进入
- 找回密码时，必须先完成邮箱验证码验证，再设置新密码

要让这两条流程真正可用，你还需要在 Supabase 控制台完成下面几项配置。

## 1. 开启邮箱确认

进入 `Authentication`，确认邮箱注册已开启，并启用 `Confirm email`。

如果这里没有开启，前端虽然会弹出验证码窗口，但 Supabase 可能会直接创建会话，无法满足“必须验证码后进入”的要求。

## 2. 配置自有 SMTP

进入 `Authentication -> SMTP Settings`，填入你自己的 SMTP 服务。

原因：

- Supabase 默认测试邮件能力不适合公开用户长期使用
- 公开注册、找回密码都依赖稳定发信
- SMTP 密钥只保存在 Supabase 后台，不会进入仁迈网页端

## 3. 修改邮件模板为 OTP 样式

需要检查这两类模板：

- Signup / Confirm signup
- Reset password / Recovery

模板里应明确展示验证码，而不是只放跳转链接。也就是说，模板内容里需要包含 Supabase 提供的 OTP token 变量。

如果你仍然保留纯链接模板，用户就会收到链接邮件，而不是仁迈网页里要求输入的验证码。

## 4. 配置 URL

进入 `Authentication -> URL Configuration`，至少加入这些地址：

- `http://127.0.0.1:8788`
- `http://localhost:8788`

后续正式部署后，还要补上：

- `https://你的-pages-dev-域名`
- `https://你的正式域名`

## 5. 当前网页端行为

配置完成后，仁迈网页端的行为会是：

1. 用户注册
2. Supabase 发送邮箱验证码
3. 网页弹出验证码窗口
4. 用户输入正确验证码
5. 验证成功后才进入仁迈

找回密码则会变成：

1. 用户点击“忘记密码”
2. Supabase 发送找回密码验证码
3. 网页弹出验证码窗口
4. 用户输入正确验证码
5. 网页要求设置新密码
6. 用户再用新密码重新登录

## 6. 当前代码对应关系

- 认证入口和验证码弹窗：`index.html`
- Supabase 注册、验证码校验、找回密码逻辑：`public-chat.js`
- 聊天数据表结构：`supabase/schema.sql`

另外，网页端现在还补了“删除当前公开账号”入口。

如果你已经在 Supabase 里执行过旧版 `schema.sql`，记得把仓库里的最新版 `schema.sql` 再执行一次，才能同步 `delete_my_account()` 这个 RPC。
